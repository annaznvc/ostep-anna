4. For the above questions, how the list is kept ordered can affect the
time it takes to find a free location for some of the policies. Use
the different free list orderings (-l ADDRSORT, -l SIZESORT+,
-l SIZESORT-) to see how the policies and the list orderings interact

    -l ADDRSORT
        Free list sorted by address (in memory order).
        Works well with FIRST fit, since nearby blocks are checked sequentially.
        Can help coalescing (adjacent blocks are next to each other).
        Search time: moderate.

    -l SIZESORT+
        Free list sorted by size (small → large).
        BEST fit becomes very fast — the first block that fits is automatically the best one.
        WORST fit becomes slow, since it must scan to the end for the largest.

    -l SIZESORT-
        Free list sorted by size (large → small).
        WORST fit becomes very fast — the first block is always the largest.
        BEST fit becomes slow, since it must scan to the smallest block that fits.


    List order	Speeds up	Slows down
    ADDRSORT	FIRST fit	BEST/WORST (if scanning entire list)
    SIZESORT+	BEST fit	WORST fit
    SIZESORT-	WORST fit	BEST fit
